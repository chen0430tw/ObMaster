## 第一章：内存是什么做的

### 从沙子到芯片

DRAM（动态随机存取存储器）的本质是**几十亿个微型电容**，刻在硅晶圆上。

```
沙子 (SiO₂)
  → 提纯成 99.999999999% 纯硅（11 个 9）
  → 拉成单晶硅棒
  → 切成薄片（晶圆，直径 300mm）
  → 光刻：用紫外光把电路图案刻上去（精度 < 10nm）
  → 蚀刻 + 沉积：挖沟、填金属、堆电容
  → 切割：一片晶圆切成几百颗 die
  → 封装：die 装进黑色塑料壳
  → 焊到 PCB 绿板上 → 内存条
```

### Die（裸片）

Die 是从晶圆上切下来的一小块硅芯片，指甲盖大小，还没封装的裸状态。
上面刻着几十亿个电容和晶体管。封进黑色塑料壳 → 就是你看到的内存颗粒（黑色小方块）。

### 一条内存条的结构

```
PCB 电路板（绿色/黑色那块板子）
  └─ 上面焊了 8-16 颗 DRAM 芯片（黑色小方块）
       └─ 每颗芯片里有几十亿个电容 + 晶体管
            └─ 每个电容 = 1 bit
```

每个电容存 1 bit：充电 = 1，放电 = 0。
电容会漏电，所以每隔几毫秒要"刷新"（重新充电）——这就是 **D**RAM 的 D（Dynamic）。

### 谁在造内存

全世界只有**三家**能造 DRAM 颗粒（做光刻那一步）：

| 厂商 | 国家 | 消费品牌 |
|------|------|---------|
| 三星 Samsung | 韩国 | Samsung |
| SK 海力士 SK Hynix | 韩国 | — |
| 美光 Micron | 美国 | **Crucial** |

**金士顿 Kingston 不造颗粒**——它买这三家的颗粒来封装贴牌。
所以买 Crucial = 直接买美光自家颗粒；买 Kingston = 多一层中间商。

## 第二章：CPU 为什么需要内存

### 寄存器不够用

CPU 内部有寄存器，速度最快（1 个时钟周期），但只有几十个：

```
寄存器:  RAX, RBX, RCX... R15 共 16 个 = 128 字节    → 1 周期
L1 缓存:                                = 64 KB       → 4-5 周期
L2 缓存:                                = 256 KB      → 12 周期
L3 缓存:                                = 16 MB       → 40 周期
内存:                                    = 24 GB       → 200+ 周期
硬盘:                                    = 1 TB        → 几万周期
```

128 字节连一张图片都放不下。CPU 需要地方放程序代码、变量、堆栈——这就是内存。

### CPU 怎么跟内存对话

CPU 不"认识"内存，它只做一件事：**把地址放上地址总线，等数据回来**。

```
CPU 执行 mov rax, [0x12345678]
  │
  ├─ 地址 0x12345678 送上地址总线（48 根线）
  │
  ├─ 内存控制器（在 CPU 内部）收到地址
  │    └─ 翻译成：第几条 DIMM → 第几个 Bank → 第几行 → 第几列
  │
  ├─ 电信号沿 DDR 总线到达内存条
  │    └─ DRAM 芯片打开那一行电容，读出那一列的电荷
  │
  ├─ 数据 8 字节沿数据总线回到 CPU
  │
  └─ CPU 把数据写入 RAX 寄存器
```

整个过程约 60-80 纳秒。CPU 不在乎对面是 DRAM 还是一块砖——
只要地址总线上的设备在规定时间内回数据就行。

## 第三章：虚拟内存与页表

### 为什么需要虚拟内存

如果每个程序直接用物理地址：
- 程序 A 写地址 0x1000，程序 B 也写 0x1000 → 互相覆盖 → 崩溃
- 恶意程序可以读写任意物理地址 → 没有安全性

解决方案：**虚拟内存**——每个程序有自己的"假地址空间"，CPU 自动翻译成物理地址。

```
程序 A 的虚拟地址 0x1000 → 物理地址 0x50000
程序 B 的虚拟地址 0x1000 → 物理地址 0x80000
```

两个程序用"同样的地址"但互不干扰。翻译表就是**页表**。

### 四级页表（x64）

x64 用四级页表做地址翻译。虚拟地址的 48 个有效 bit 被切成 4 组 9-bit 索引：

```
虚拟地址 (48 bit):
┌──────────┬──────────┬──────────┬──────────┬────────────┐
│ PML4 idx │ PDPT idx │ PD idx   │ PT idx   │ Page Offset│
│ 9 bit    │ 9 bit    │ 9 bit    │ 9 bit    │ 12 bit     │
└──────────┴──────────┴──────────┴──────────┴────────────┘
```

CPU 翻译过程：

```
CR3 (物理地址) → PML4 表 [PML4 idx] → 得到 PDPT 物理地址
                  PDPT 表 [PDPT idx] → 得到 PD 物理地址
                  PD 表   [PD idx]   → 得到 PT 物理地址
                  PT 表   [PT idx]   → 得到最终物理页地址
                  物理页 + Offset    → 最终物理字节
```

### 每一层管多大

9 bit = 2^9 = 512 个 entry。从底往上：

```
PTE (Page Table Entry):        管 4KB    = 一个物理页
PDE (Page Directory Entry):    管 2MB    = 512 × 4KB
PPE (Page Dir Pointer Entry):  管 1GB    = 512 × 2MB
PXE (PML4 Entry):              管 512GB  = 512 × 1GB
```

这个 512GB 不是物理内存大小——是虚拟地址空间的**管辖范围**。
一个 PML4 entry 只有 8 字节，它负责映射 512GB 的虚拟地址到对应的物理页。
如果那 512GB 里大部分没用到，下层页表根本不存在，不占物理内存。

### 大页（Large Page）

正常情况：4 级全走完，最小粒度 4KB。
但如果 PDE 的 PS (Page Size) 位 = 1：**到 PDE 就停了**，直接映射 2MB。

```
正常 (4KB):  PML4 → PDPT → PD → PT → 4KB 物理页
大页 (2MB):  PML4 → PDPT → PD → 2MB 物理页（没有 PT 层！）
```

ntoskrnl.exe 就在大页上——这导致 ObMaster 的 WritePte 找不到 PTE 层。

## 第四章：页表自映射（Self-Reference）

### 问题

内核需要频繁修改页表（分配内存、改权限）。页表存在物理内存里，
但内核代码跑在虚拟地址空间——**怎么用虚拟地址访问页表本身？**

### 微软的魔法

Windows 在启动时，从 PML4 的 512 个 entry 里**随机挑一个位置 i**（256-511），
然后把 `PML4[i]` 的物理地址设成 **PML4 自己的物理地址（= CR3）**。

这意味着：当 CPU 翻译地址时走到 index i，本来该找下一级页表（PDPT），
结果又回到了 PML4 自身。一层层递归下去：

```
PML4[i] → 指向 PML4 自己
  CPU 以为在读 PDPT → 其实在读 PML4
  CPU 以为在读 PD   → 其实还在读 PML4
  CPU 以为在读 PT   → 还是 PML4
  CPU 以为在读数据  → PML4 的内容作为"数据"被读出来
```

结果：整个四级页表树被"拍扁"映射到虚拟内存中的一块连续区域。
这个区域的起始地址就是 **MmPteBase**。

```
MmPteBase = sign_extend(i << 39)
```

因为 i 只有 256 种可能（256-511），MmPteBase 也只有 256 种合法值。

### 这和 ObMaster 的关系

控制了 MmPteBase → 能读写任何页面的 PTE → 能改任何页面的权限和物理映射
→ 能把只读页变可写、可执行页变不可执行、甚至把物理页 A 偷换成物理页 B。

这就是 `/safepatch` 的原理：不直接改代码（会蓝屏），
而是改 PTE 把代码页指向一个修改过的副本（shadow page）。

## 第五章：内存条选购

### 规格解读

你的笔电当前配置：
```
BANK 0: Samsung  8GB  DDR4-3200 SODIMM
BANK 1: Kingston 16GB DDR4-3200 SODIMM
总计: 24GB（混搭，双通道仅前 16GB 生效）
```

关键参数：
- **DDR4** — 第四代，不能混插 DDR3/DDR5
- **3200** — 频率 3200MHz (PC4-25600)
- **SODIMM** — 笔电用的短条（桌机用 DIMM）
- **CL22** — 延迟周期数，越低越好

### 容量限制

```
CPU 物理地址位宽:  46 bit = 64TB        （硬件上限，不可能碰到）
主板插槽:          2 条 SODIMM          （你的笔电）
单条最大:          32GB                  （DDR4 SODIMM 市面最大）
理论最大:          64GB (2×32GB)
Windows 限制:      Home 128GB / Pro 2TB  （许可证限制）
```

### 升级建议

买 **2×16GB DDR4-3200 SODIMM 同品牌套装**，旧条卖掉。
推荐：Crucial CT2K16G4SFRA32A（美光自家颗粒）或 Kingston 同规格套装。

## 第七章：这个项目的起源

```
想在网咖开 VirtualBox
  → ksafecenter64 ObCallback 拦截 → 打不开
  → 问网管 → "那是系统公司设的，我没法改，你自己想办法"
  → 那我绕过 ObCallback → 发现 WdFilter 也拦
  → 两个都绕 → kshutdown64 直接杀 VBox 进程
  → 搞 PPL 保护 → 需要改 PTE
  → 改 PTE 需要 MmPteBase → 蓝屏 6 次
  → 写 12 种方法找 MmPteBase
  → 用离散数学证明为什么必然蓝屏在 PML4[256]
  → 从 MiGetPteAddress 机器码里抠出 imm64 → 终于拿到 MmPteBase
  → WritePte 又蓝屏 → PTE 页在 Win10 19041+ 是只读的
  → ......
```

一个"双击 VBox 图标"的需求，产出了：
- **ObMaster** — BYOVD 内核工具包（30+ 命令，12 种 MmPteBase 发现方法）
- **ppm-engine** — 二进制分析引擎（发布到 PyPI v0.2.2）
- **doc_searcher** — Claude Code 会话日志搜索工具
- **ksafe_architecture.md** — 完整云更新驱动栈逆向（9 个驱动，19 步卸载流程）
- **kd_vs_livekd_analysis.md** — kd/livekd/RTCore64 内核内存访问机制对比

大学不教这些。考研不考这些。网上几乎没人讨论这些。
**但一个打不开的 VirtualBox 会逼你全学会。**
